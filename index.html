How 0/1 Knapsack Works Dynamic Programming
1
The Problem
You have a bag with a weight limit W and n items, each with a weight and value. Pick items to maximise total value without exceeding W. Each item is either fully taken (1) or left behind (0) — no fractions.
2
Build a Table
Create a grid: rows = items, columns = capacities 0 to W. Cell matrix[i][cap] answers: "What's the best value using only items 1 to i, with a bag of size cap?" Start everything at 0.
3
Fill It Row by Row
For each item and each capacity, make a binary decision:
✗ Exclude — item too heavy, or keeping it out is better → copy the value from the row above.
✓ Include — item fits and adding it improves the total → take the best of (exclude vs include).
4
Read the Answer
The bottom-right cell matrix[n][W] holds the maximum achievable value. To find which items were chosen, trace back through the table from bottom-right to top-left — if a row's value changed, that item was included.
5
Why Not Just Try Everything?
With n items there are 2ⁿ possible subsets. At n=20 that's over 1 million to check. DP reuses answers — each cell is computed once and looked up later. Total work: just n × W cells.

Each cell matrix[i][cap] is computed exactly once and stored. When later cells need it, they look it up instead of recomputing. Overview — What is 0/1 Knapsack & how does DP solve it? ▲

The problem: You have a bag of capacity W and n items each with a weight wᵢ and value vᵢ. Choose a subset that maximises total value without exceeding W. Each item is taken entirely (1) or not at all (0) — you cannot take a fraction.

Why greedy fails: Sorting by value/weight ratio and picking greedily can block a better combination. Example: W=5, items {v=6,w=1}, {v=10,w=2}, {v=12,w=3}. Greedy picks item 1 + item 2 (value=16). DP finds item 2 + item 3 (value=22).

How DP works: Build a table matrix[i][cap] = "best value using items 1..i with capacity cap". For each cell, make a binary decision: exclude item i (copy row above) or include it (add vᵢ to best value at cap−wᵢ). Traceback from matrix[n][W] to recover which items were chosen.

When to use it: Any problem requiring optimal selection from a fixed set of items with a constraint — budget allocation, cargo loading, project selection, portfolio optimisation.

O(nW) time — pseudo-polynomial, fast when W is moderate
Guaranteed optimal — DP always finds the true best answer
O(nW) space — reducible to O(W) using a 2-row rolling array
Worked Example — Classic preset step by step ▲

Setup: W = 8, Items: {v=1,w=2}, {v=2,w=3}, {v=5,w=4}, {v=6,w=5}

1
Initialise — Row 0 and Column 0 = 0
With zero items or zero capacity, we can pack nothing. Every cell in row 0 and column 0 is set to 0. These are the base cases the recurrence builds on.
2
Row 1 — Item 1 (v=1, w=2)
For cap 0–1: item weighs 2, doesn't fit → copy 0 from row above. For cap 2–8: item fits. max(exclude=0, include=0+1=1) = 1. Row 1 becomes [0,0,1,1,1,1,1,1,1].
3
Row 2 — Item 2 (v=2, w=3)
At cap=5: max(exclude=1, include=matrix[1][2]+2=1+2=3) = 3. At cap=8: max(exclude=1, include=matrix[1][5]+2=1+2=3) = 3. Row 2 shows value 3 appears at cap ≥ 5.
4
Row 3 — Item 3 (v=5, w=4)
At cap=4: max(0, 0+5=5) = 5. At cap=7: max(3, matrix[2][3]+5=2+5=7) = 7. At cap=8: max(3, matrix[2][4]+5=1+5=6) → 7 wins from previous. Row ends at 7.
5
Row 4 — Item 4 (v=6, w=5)
At cap=8: max(exclude=7, include=matrix[3][3]+6=2+6=8) = 8. This is our answer — matrix[4][8] = 8.
6
Traceback — which items were chosen?
matrix[4][8]=8 ≠ matrix[3][8]=7 → Item 4 included. Move to cap 8−5=3. matrix[3][3]=2 = matrix[2][3]=2 → Item 3 excluded. matrix[2][3]=2 ≠ matrix[1][3]=1 → Item 2 included. Move to cap 3−3=0. Done. Chosen: Item 2 + Item 4 = value 8, weight 8.
Key Concepts & Common Mistakes ▲
📌 Overlapping Subproblems
The same sub-question "what's the best value using items 1..i at capacity c?" appears many times. DP stores answers in the table instead of recomputing — this is the core insight that makes it efficient.
📌 Optimal Substructure
The optimal solution to the full problem contains optimal solutions to sub-problems. matrix[i][w] is built entirely from already-optimal smaller cells — the recurrence is valid because of this property.
⚠ Common Mistake: Greedy Assumption
Students often assume sorting by ratio and picking greedily is optimal. It works for Fractional Knapsack but NOT 0/1 Knapsack. Use the 🔴 Greedy Fails preset to see a concrete counterexample.
⚠ Common Mistake: Index Off-by-One
The table is (n+1) × (W+1). Row 0 and column 0 are base cases. Item i is stored in row i. When tracing back, compare matrix[i][w] with matrix[i−1][w] — NOT matrix[i][w−1].
📌 Pseudo-Polynomial Time
O(nW) looks polynomial but W is a numeric value, not the size of the input encoding. If W = 2³⁰, the table has billions of cells. This is why 0/1 Knapsack is NP-hard despite having an "efficient" DP.
📌 Traceback vs. Table Fill
Filling the table (forward pass) finds the optimal VALUE. Traceback (backward pass) finds WHICH ITEMS produce that value. Both are required for a complete solution. The traceback starts at matrix[n][W] and works up-left.

01 Problem Description

The 0/1 Knapsack problem asks: given n items, each with a weight wᵢ and value vᵢ, and a knapsack of capacity W, select a subset of items that maximises total value without exceeding total weight W. Each item is either taken entirely or not at all — hence "0/1".

A brute-force approach checks all 2ⁿ subsets. Dynamic Programming solves it in O(nW) time using a table K[i, w] representing the maximum value achievable using items 1..i with capacity w.

Recurrence
if wᵢ ≤ w: K[i, w] = max( K[i−1, w], vᵢ + K[i−1, w−wᵢ] )
else: K[i, w] = K[i−1, w]
Reference Cormen et al., Introduction to Algorithms, Chapter 16 (greedy comparison). See also: wikipedia.org/wiki/Knapsack_problem

CLO-1 Worst Case — Why Large n and W Is the Hardest Input

For 0/1 Knapsack DP, there is no best or average case in the traditional sense — the algorithm always fills every cell of the (n+1)×(W+1) table regardless of item values. The only factor that determines how slow it runs is how large n and W are.

What makes the worst case?
  • Maximum number of items n = 7 (the ⚠ Worst Case preset)
  • Maximum capacity W = 15
  • This forces the algorithm to fill 7 × 16 = 112 cells
  • Every cell requires 1–2 lookups and 1 comparison
  • Total operations: ~336–448
Compare against brute force:
  • Brute force checks 2⁷ = 128 subsets
  • Each subset takes O(n) = 7 operations to verify
  • Total: 896 operations
  • At n=20: DP = 20×W vs brute force = 20,971,520
  • DP wins by orders of magnitude

Why Greedy Fails for 0/1 Knapsack

The greedy strategy sorts items by value/weight ratio and picks highest-ratio items first. This is optimal for Fractional Knapsack — but for 0/1 Knapsack, taking a high-ratio item early can leave a gap no remaining item can fill, giving a suboptimal total.

CLO-2 Sorting Step — Its Role and What Happens Without It
Role of sorting in this algorithm:

The greedy strategy depends entirely on sorting items by value/weight ratio before making any decisions. This sort defines the order of consideration — highest ratio first. It is highlighted in orange in the bar chart above when you click Animate Sort.

The DP algorithm also uses a form of "search" during its traceback phase — it scans the filled table from bottom-right to top-left to recover which items were chosen (purple cells in the Visualizer animation).

What changes without the sort?
  • Without sorting: greedy picks items in original input order — almost certainly suboptimal
  • Without traceback: you get the optimal value but cannot recover which items to pack
  • Greedy sort is O(n log n) — the most expensive step in the greedy approach
  • DP traceback is O(n) — linear scan of one column of the table
CLO-3 Design Paradigm — Dynamic Programming
Which paradigm and why:

0/1 Knapsack uses Dynamic Programming. The problem has two key DP properties:

  • Overlapping subproblems: matrix[i][w] is used by multiple later cells — computed once, reused many times
  • Optimal substructure: the optimal answer for (items 1..i, capacity w) is built from optimal answers to smaller sub-problems

Go to the Visualizer and watch the DP table fill — each new cell reuses already-computed cells from the row above. This is overlapping subproblems in action.

Why other paradigms fail:
  • Greedy (wrong): the example below shows greedy gives value — while DP gives — — load 🔴 Greedy Fails preset to see this
  • Divide & Conquer (wrong): D&C splits the problem but sub-problems overlap — it would recompute the same cells exponentially many times without memoisation, reducing to brute force
  • Brute Force (correct but slow): checks all 2ⁿ subsets — correct but O(n·2ⁿ)

0/1 vs Fractional Knapsack

One small change transforms the problem: allow fractions. With fractions, greedy is provably optimal. Without fractions (0/1), greedy can fail and DP is required. Same items, same capacity, completely different rules.

FFractional: greedy wins
010/1: DP required
!Greedy fails for 0/1
✓ Fractional Knapsack
  • Take any fraction of any item
  • Greedy by ratio is provably optimal
  • Time complexity: O(n log n)
  • No DP table needed
  • Examples: liquid cargo, gold bars, oil barrels
0/1 Knapsack
  • Whole items only — take (1) or leave (0)
  • DP gives guaranteed optimal answer
  • Time complexity: O(n × W)
  • Greedy can fail — see Greedy vs DP tab
  • Examples: budgets, containers, project selection
